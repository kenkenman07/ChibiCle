# 青切符ドライブ 設計書

## 1. アプリ概要

自転車走行中に **GPS** と **カメラ映像** をリアルタイムで取得し、
「信号無視」「一時不停止」「右側通行」を自動検知して走行ログと違反情報を保存・表示する **PWA（Progressive Web App）**。

スマートフォンのブラウザ上で動作し、ホーム画面に追加することでネイティブアプリに近い体験を提供する。
2026 年 4 月施行の自転車青切符制度に対応。

---

## 2. 技術スタック

### フロントエンド（PWA）

| 項目 | 選定技術 | 理由 |
|------|----------|------|
| フレームワーク | **React 19 + Vite 6** | 高速 HMR・バンドル最適化 |
| 型安全 | **TypeScript 5.7** | 全コードベースで型チェック |
| PWA | **vite-plugin-pwa**（`generateSW` モード） | Service Worker + Manifest 自動生成 |
| HTTPS | **@vitejs/plugin-basic-ssl** | ローカル開発でも GPS/カメラ動作に必要 |
| カメラ | `MediaDevices.getUserMedia()` + **Canvas toBlob** | フレーム単位の JPEG 送信に最適 |
| GPS | `navigator.geolocation.watchPosition()` | リアルタイム位置追跡 |
| 画面スリープ防止 | **Screen Wake Lock API** | 走行中の画面 OFF 防止 |
| 地図表示 | **Leaflet.js** (`react-leaflet`) | OSS・軽量、OSM タイルを無料利用可 |
| 通信 | **fetch** (API クライアント層経由) + 生 **WebSocket** | 軽量。外部ライブラリ不要 |
| 状態管理 | **Zustand** | 走行中のリアルタイム状態を一元管理 |
| オフラインストレージ | **Dexie.js**（IndexedDB ラッパー） | GPS ポイントのオフラインバッファリング |
| UI | **Tailwind CSS 3** + **lucide-react** | モバイルファーストの UI + アイコン |
| ルーティング | **React Router v7** | SPA ルーティング |

> [!WARNING]
> **ブラウザ制約（PWA の限界）**
> - GPS・カメラは **HTTPS 必須**（localhost は例外）
> - **バックグラウンド GPS 取得は不可能**：Wake Lock API で画面 ON を維持する設計が必須
> - iOS Safari は SW がバックグラウンドで即座にフリーズ。Background Sync も非対応
> - iOS でカメラを使うには `<video>` 要素に `playsInline` 属性が必須

### バックエンド

**Clean Architecture + Strategy パターン** を採用。

| 項目 | 選定技術 | 理由 |
|------|----------|------|
| フレームワーク | **FastAPI** | 非同期処理・WebSocket・自動 OpenAPI スキーマ生成 |
| 画像解析 | **YOLOv8n** (Ultralytics) | 軽量モデルで信号機検出（COCO 事前学習） |
| 信号色判定 | **OpenCV** | 検出した信号機 ROI の HSV 色空間解析 |
| 地図データ | **OpenStreetMap + Overpass API** | 一時停止標識・道路ジオメトリの取得 |
| 位置計算 | **geopy** | GPS 座標間の測地線距離計算 |
| バリデーション | **Pydantic v2 + pydantic-settings** | 型安全なリクエスト/レスポンス + 環境変数管理 |
| HTTP クライアント | **httpx** | Overpass API への非同期リクエスト |

### ストレージ（現在）

| 項目 | 実装 | 備考 |
|------|------|------|
| フロントエンド | **IndexedDB (Dexie.js)** | GPS・Trip・Violation のローカル保存（主ストレージ） |
| バックエンド | **InMemory dict** | MVP 用。将来 Supabase に差替可能（Repository 抽象化済） |

### バックエンド設計パターン

|----------|----------|

| パターン | 適用箇所 |
|----------|----------|
| **Facade** | Custom Hooks (`useGpsTracker`, `useCameraStream`, `useWakeLock`) がブラウザ API の複雑さを隠蔽 |

| **Observer** | Dexie `useLiveQuery` で IndexedDB の変更をリアクティブに UI 反映 |
| **Local-First** | IndexedDB が主ストレージ、バックエンド同期は best-effort |

## 5. 違反検知ロジック

### 5-1. 信号無視 (`signal_ignore`) — カメラ解析

```
JPEG (2fps, WebSocket) → YOLOv8n → 信号機 ROI → HSV 赤色判定 → 速度チェック → 違反
```

| ステップ | 処理 | 閾値 |
|----------|------|------|
| 速度フィルタ | 速度 < 5 km/h → スキップ（停車中は無視） | `5.0 km/h` |
| 物体検出 | YOLOv8n (COCO class 9 = traffic light) | 信頼度 `0.4` |
| ROI 切出し | バウンディングボックスで信号機領域を抽出 | — |
| 信号機方向判定 | ROI のアスペクト比で横型/縦型を判別 | `width > height` → 横型 |
| 赤色チェック領域 | **横型**: 左 1/3（日本の標準: 左=赤, 中=黄, 右=青）/ **縦型**: 上 1/3 | — |
| HSV 赤色範囲 | 低域: H 0-10, S 70-255, V 50-255 / 高域: H 170-180, S 70-255, V 50-255 | — |
| 赤色比閾値 | 赤ピクセルがチェック領域面積の 15% 超 → 赤信号 | `15%` |

**YOLO 推論は `asyncio.to_thread` でスレッドプールに委譲**し、FastAPI のイベントループをブロックしない。

### 5-2. 一時不停止 (`no_stop`) — GPS 解析

```
GPS バッチ (5s) → Overpass API で一時停止標識取得 → 10m 以内? → ±5 点で停止? → 違反
```

| ステップ | 処理 | 閾値 |
|----------|------|------|
| 精度フィルタ | accuracy > 20m → スキップ | `20 m` |
| 標識検索 | Overpass API `highway=stop` ノード（500m グリッドキャッシュ, 24h TTL） | `500 m` |
| 距離判定 | GPS ポイントと標識の測地線距離（geopy） | `10 m` |
| 停止判定 | 前後 ±5 ポイント窓内に速度 < 3 km/h が **1 つでも**あれば停止とみなす | `3.0 km/h` |
| 違反判定 | 窓内に停止ポイントが 0 件 → 違反 | — |
| 重複防止 | 同一標識（osm_id）は 1 バッチ内で 1 回のみ判定 | — |

### 5-3. 右側通行 (`right_side_riding`) — GPS + 道路ジオメトリ解析

```
GPS 履歴 → OSM 道路 way 取得 → 各ポイントで左右判定 → スライディングウィンドウ投票 → 違反
```

| ステップ | 処理 | 閾値 |
|----------|------|------|
| フィルタ | accuracy > 20m or 速度 < 5 km/h → スキップ | `20 m` / `5.0 km/h` |
| 道路取得 | Overpass API で `trunk`〜`cycleway` の way ジオメトリ（`out body geom`） | `500 m` 圏 |
| 最近傍道路 | 各 GPS ポイントから全道路セグメントへのフラットアース近似距離 | `30 m` 以内 |
| 進行方向 | 連続 2 点から自転車の方位角（bearing）を算出 | — |
| 道路方位 | 最近傍道路セグメント A→B の方位角を算出 | — |
| 同方向判定 | bearing 差 < 90° → 同方向 | `90°` |
| 左右判定 | 外積 `(B-A) × (P-A)` の符号（正=左, 負=右） | — |
| 日本ルール | 同方向なら左側, 逆方向なら右側にいるべき → `is_on_left != same_direction` で違反 | — |
| 一方通行 | `oneway=yes` で逆方向 → 自動的に違反 | — |
| ウィンドウ | 直近 20 ポイント中、判定可能点の **70% 以上**が右側 → 違反 | `20 点` / `70%` |
| 最低判定数 | 判定可能ポイント < 5 → 判定しない | `5 点` |
| クールダウン | 同一 trip・同一違反タイプは 120 秒間隔でのみ発火 | `120 s` |

**左右判定の数学的根拠:**

```
外積 (B-A) × (P-A) > 0  →  P は A→B の左側
外積 (B-A) × (P-A) < 0  →  P は A→B の右側

日本は左側通行:
  自転車が道路と同方向 → 左側にいるべき → 外積が正なら OK
  自転車が道路と逆方向 → 右側にいるべき → 外積が負なら OK
  violation = (is_on_left != same_direction)
```

## 7. API 設計

### GPS 解析（HTTP）

```
POST /api/gps
Content-Type: application/json

{
  "trip_id": "uuid",
  "points": [
    { "lat": 35.123, "lng": 135.456, "speed_kmh": 12.3, "accuracy_m": 5.2, "recorded_at": "..." }
  ]
}

→ 200 OK
{
  "saved": 5,
  "violations": [
    { "type": "no_stop", "lat": 35.120, "lng": 135.450, "detected_at": "..." },
    { "type": "right_side_riding", "lat": 35.121, "lng": 135.451, "detected_at": "..." }
  ]
}
```

### カメラ解析（WebSocket）

```
WS /ws/camera?trip_id=xxx

Client → Server: バイナリフレーム (JPEG, 640x480, 2 fps)
Server → Client: JSON

// 赤信号 + 速度超過 → 違反イベント
{ "type": "violation", "data": { "violation_type": "signal_ignore", "lat": ..., "lng": ..., "detected_at": "..." } }

// ハートビート
Server → { "type": "ping" }
Client → { "type": "pong" }
```

**バックプレッシャー**: `asyncio.Queue(maxsize=100)`。キュー満杯時は最古のフレームを破棄。
**ハートビート**: 30 秒間隔で ping。5 秒以内に pong がなければ接続クローズ。

### 走行セッション（HTTP）

```
POST   /api/trips              → 走行開始（trip 作成）
PATCH  /api/trips/:id/end      → 走行終了（ended_at 更新）
GET    /api/trips/:id          → 走行結果取得
GET    /api/trips              → 過去走行一覧
GET    /health                 → ヘルスチェック
```

## 8. フロントエンド/バックエンド分離

// apiFetch(path, init) — VITE_API_BASE_URL を prefix して fetch

| OSM タイル | **CacheFirst** | maxEntries: 500, 有効期限 7 日 |
| Leaflet アセット | **CacheFirst** | maxEntries: 20, 有効期限 30 日 |

`registerType: 'autoUpdate'` により、新バージョンは自動適用。

### 走行中画面の状態遷移

```
[走行開始（RidingPage マウント時に自動開始）]
    │
    ├─ Trip を IndexedDB に作成
    ├─ Trip を POST /api/trips で作成 (best-effort)
    ├─ Wake Lock 取得
    ├─ GPS watchPosition 開始
    ├─ カメラ getUserMedia + WebSocket 接続
    ├─ 1 秒タイマー開始
    │
    ▼
[走行中]
    ├─ GPS ポイント → Dexie に即時保存 → 5 秒バッチでバックエンド同期
    ├─ カメラフレーム → Canvas → WebSocket 送信 (2 fps)
    ├─ サーバーから違反通知受信 → Dexie 書込 + HUD 更新
    │
[走行終了ボタン押下]
    │
    ├─ GPS clearWatch
    ├─ カメラ track.stop() + WebSocket close
    ├─ Wake Lock 解放
    ├─ 未同期 GPS を最終フラッシュ
    ├─ PATCH /api/trips/:id/end (best-effort)
    │
    ▼
[結果画面へ遷移]
```

## 12. 主要リスクと対策

| リスク | 対策 |
|--------|------|
| バックグラウンド GPS 停止 | Wake Lock API で画面 ON 維持 |
| YOLOv8 のリアルタイム処理が重い | `asyncio.to_thread` でスレッドプール委譲、2 fps に制限 |
| GPS 精度が停止/左右判定に不十分 | accuracy > 20m フィルタ、停止判定 10m、右側通行は 70% ウィンドウで誤検知抑制 |
| WebSocket 接続不安定（モバイル） | カメラ WS エラー時はローカルキャプチャのみ継続。GPS は HTTP バッチで独立 |
| iOS PWA の制限 | Background Sync 非対応 → `online` / `visibilitychange` イベントで手動同期 |
| OSM 一時停止標識の漏れ | 500m グリッドキャッシュ (24h TTL) で API 負荷軽減。将来: ユーザー報告で補完 |
